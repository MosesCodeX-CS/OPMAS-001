# OPMAS Setup Guide

This repository has two main runtime parts:

- `opmas-app/` — the Laravel dashboard and web UI
- `collector/` — the Python Modbus PLC telemetry collector and simulator service

They each have their own environment file:

- `opmas-app/.env` — Laravel web app settings, including the application key and the database connection for the dashboard
- `collector/.env` — Modbus PLC connection settings and the collector database connection settings

## 1) Prerequisites

Before starting, make sure the Python collector environment is already prepared inside the `collector/` folder.

```bash
cd collector
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

You will also need:

- PHP 8.2+
- Composer
- A working Laravel database connection

## 2) Laravel app setup

The Laravel app needs its own `.env` file at `opmas-app/.env`.

At minimum, that file must provide:

- `APP_KEY` — generated by Laravel with `php artisan key:generate`
- a working database connection for `opmas-app`
- the usual Laravel app settings such as `APP_URL`

For a local SQLite setup, use the following pattern in `opmas-app/.env`:

```env
APP_NAME=OPMAS
APP_ENV=local
APP_KEY=
APP_DEBUG=true
APP_URL=http://127.0.0.1:8000

DB_CONNECTION=sqlite
DB_DATABASE=/home/mosisto/opmas/opmas-app/database/database.sqlite
```

If you are using MySQL, the Laravel app `.env` should look like this:

```env
APP_NAME=OPMAS
APP_ENV=local
APP_KEY=
APP_DEBUG=true
APP_URL=http://127.0.0.1:8000

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=db_name
DB_USERNAME=db_username
DB_PASSWORD=db_password
```

Then run:

```bash
cd opmas-app
composer install
cp .env.example .env
php artisan key:generate
php artisan migrate --seed
php artisan serve --host 127.0.0.1 --port 8000
```

The `php artisan key:generate` command creates the Laravel application key and writes it into `APP_KEY` inside `opmas-app/.env`.

Then open the dashboard at:

- http://127.0.0.1:8000

### Default login accounts

The seed data creates these users:

- `admin@example.com` / `admin_password`
- `user@example.com` / `user_password`
- `system_admin@example.com` / `system_admin_password`

## 3) Collector setup

The OPMAS telemetry collector runs as a background `systemd` service. For local development, there is also a virtual Modbus TCP PLC simulator service.

### 3.1 Configure environment variables

Inside the `collector/` folder:

```bash
cp .env.example .env
```

Update the `collector/.env` file with your local Modbus PLC and collector database settings.

Typical local values for a SQLite-backed local development setup:

```env
PLC_HOST=127.0.0.1
PLC_PORT=5020
POLL_INTERVAL=5

DB_DRIVER=sqlite
DB_PATH=/home/mosisto/opmas/opmas-app/database/database.sqlite
```

If you are using MySQL instead, set the MySQL values in the same file and point `DB_DRIVER=mysql` to the matching database host, port, username, password, and name.

### 3.2 Install and start the services

The repository includes a service installer script that detects the local paths and configures the `systemd` services automatically.

```bash
cd collector
chmod +x services/setup_services.sh
./services/setup_services.sh
```

This script will:

- copy the service definitions into `/etc/systemd/system/`
- reload `systemd`
- enable the services for boot
- start the collector and simulator services

## 4) Useful service management commands

### Check service status

```bash
sudo systemctl status opmas-collector
sudo systemctl status opmas-simulator
```

### View logs

```bash
sudo journalctl -u opmas-collector -f
sudo journalctl -u opmas-simulator -f
```

### Stop the services

```bash
sudo systemctl stop opmas-collector
sudo systemctl stop opmas-simulator
```

### Restart the services

```bash
sudo systemctl restart opmas-collector
sudo systemctl restart opmas-simulator
```

## 5) Running the Collector & Simulator (Development Options)

You can run the collector and simulator in one of two ways. Choose **only one** option to avoid port/database conflicts:

### Option A: Background Services (Recommended for Linux)
If you installed the services using the `setup_services.sh` script, systemd runs them automatically in the background. You do not need to run manual terminal commands.
* **To check if they are running:**
  ```bash
  sudo systemctl status opmas-collector
  sudo systemctl status opmas-simulator
  ```

### Option B: Manual Terminal Execution (For Debugging / macOS / Windows)
If you prefer to see print statements directly in your console or are developing on a platform without systemd, stop the background services and run them in separate terminal windows:

1. **Stop systemd services (if active):**
   ```bash
   sudo systemctl stop opmas-collector opmas-simulator
   ```
2. **Terminal 1 — Modbus PLC Simulator:**
   ```bash
   cd collector
   source venv/bin/activate
   python3 simulator.py
   ```
3. **Terminal 2 — Telemetry Collector Daemon:**
   ```bash
   cd collector
   source venv/bin/activate
   python3 main.py
   ```
4. **Terminal 3 — Laravel Web UI:**
   ```bash
   cd opmas-app
   php artisan serve --host 127.0.0.1 --port 8000
   ```

The dashboard and collector are configured independently:
* `opmas-app/.env` controls the Laravel web application.
* `collector/.env` controls the PLC polling registers and the telemetry destination database.

## 6) Troubleshooting

### Laravel app does not load

- Confirm `opmas-app/.env` exists
- Run `php artisan key:generate`
- Run `php artisan migrate --seed`
- Confirm the port `8000` is not already in use

### Collector cannot connect

- Confirm `collector/.env` has the correct PLC host and port
- Make sure the collector’s database path or MySQL settings are correct
- Check the `systemd` logs with `journalctl`

## 7) Notes

- This project is primarily configured for local development and service-based collector operation.
- The collector is the data feeder into the Laravel app, while the dashboard is the web visualization layer.
